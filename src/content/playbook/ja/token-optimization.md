## 一言で

<div class="hero-quote">
  <p>
    <strong>トークンを数えるな、1 トークンを意味あるものにせよ。</strong>
  </p>
  <p>
    最大のトークン削減は <strong>エージェントの品質</strong> から生まれる。失敗ループが減り、誤ったファイルを触らなくなり、デバッグの往復が消える。
  </p>
</div>

> ⚠️ 間違った方向に走っているエージェントの **プロンプトを短くする** のは純粋なムダ。**まず品質を上げる、コストは副作用として下がる。**

## なぜ「品質」が一番のレバーなのか

### Agent ROI

```
Agent ROI =（エージェント出力の価値 − トークンコスト）/ トークンコスト
```

価値が **ゼロ**（間違ったファイル・誤った修正・幻覚）のとき、いくらトークンを削っても ROI は出ない。分子を動かせるのは品質だけ。

### 複利でエラーが効いてくる（Compound Error Problem）

LLM は非決定的。1 ステップの小さな誤差率が、多ステップワークフロー全体では大きく効いてくる。

| 1 ステップの正確さ | 10 ステップ後 | 50 ステップ後 |
| --- | --- | --- |
| 99% | 90% | **60%** |
| 95% | 60% | **8%** |
| 90% | 35% | **0.5%** |

失敗ループ 1 回は **そのループ自体 + 修正ループ + 人間のクリーンアップ時間** を全部食う。最速のトークン削減策は「リトライを発生させない」こと。

> 🚀 *NASA が 20 発のロケットを月にだいたい向けて打ち、1 発でも当たれと祈る ── 今のエージェント運用は割とこれ。トークンが安かった時代はギャンブルでよかった。今は違う。*

## メンタルモデル

### LLM — ステートレスな確率機械

テキスト入力 → テキスト出力。内部では「次に来る確率の高いトークン」を予測しているだけ。**呼び出し間に記憶を持たない**。セッションで「知っていること」は毎回すべて送り直す必要がある。

> 💰 **入力トークン vs 出力トークン** — 出力は入力より高い（生成側に GPU の計算コストが乗るから）。「**簡潔に**」と一言入れるだけで、品質に影響せずに実費が下がる。

### エージェント（ハーネス）— ステートフルなループ

エージェント（VS Code Chat / Copilot CLI / Cloud Agent / Claude Code / Codex …）は、あなたの代わりに **LLM と何度もやり取り** する。毎ループ送られているのは：

- **システムプロンプト + ツール定義**（常時）
- あなたの **永続インストラクション**（常時）
- **プロンプトとこれまでの応答**（成長していく）
- セッション中に読まれた **ファイル群**

> 🧠 マジックはない、ぜんぶテキスト。ハーネスもまた text-in / text-out で、毎ターン LLM に見せる内容を組み立てているだけ。

> 🗂️ **キャッシュは保証なし。** 同じプレフィックスがあれば LLM 側で再利用されることはあるが、TTL は短く（数分程度のモデルもある）。コーヒー休憩から戻ったら消えていることも。**キャッシュを当てにした最適化はしない。** パワーユーザーが詰めるための領域で、初手で狙う場所ではない。

### Context Rot

コンテキストウィンドウを埋めるほど賢くなる、わけではない。よく知られた失敗モードが 2 つ：

- **Lost in the Middle** — 50% 以上埋まると、真ん中の内容が無視されやすくなる。途中でタスクを切り替えると、序盤のバグ修正に戻ろうとし続ける、というのが典型。
- **Recency Bias** — 末尾の内容が重要さに関係なく重視される。システムプロンプトや非交渉事項を忘れてエージェントが暴走しがち。

> 🎯 **対処の目安：60〜70% を超えたら警告サイン。** 世界の終わりではないが、**`/clear` で仕切り直す** か、**計画だけを抽出して新セッションに渡す** タイミング。タスクが変わったら必ず新セッション。

> 📘 詳細は <a class="retro-link" href="/theomonfort/playbook/context-engineering">Context Engineering ↗</a> ページ（treemap、何が窓を埋めるか、対処）。参照: <a href="https://www.producttalk.org/context-rot" target="_blank" rel="noopener noreferrer" class="retro-link">Context Rot — Product Talk</a>。

## なぜ「運用」カテゴリにあるのか

<div class="hero-quote">
  <p>
    これらの最適化は、<strong>1 つの同期エージェント</strong> を使う段階から <strong>複数の非同期エージェントを束ねる</strong> 段階に移るほど重要になる。
  </p>
</div>

1 日に数回エージェントを叩く程度なら、ムダはだいたい許容範囲。Cloud Agent タスク・Agentic Workflow・CLI セッションを **並列で何十本** 走らせ始めた瞬間、すべての非効率は掛け算で効いてくる。だからトークン最適化は Metrics や Memory と同じく **運用の練習** として扱う。

## 2 つの最大のレバー

### 1 — モデル選び

| ティア | 例 | 用途 |
| --- | --- | --- |
| 🧠 **Reasoning** | Claude Opus 4.7 / GPT-5.5 | 同期の設計・アーキ・デバッグ・難しいレビュー |
| ⚡ **Mid** | Claude Sonnet / GPT-5.4 | 非同期の実装、ほとんどの Cloud Agent タスク |
| 🪶 **Low** | Claude Haiku / GPT-5 mini | 小さなリファクタ、繰り返し、ドキュメント更新 |
| 🤖 **Auto** | — | ものぐさのデフォルト ── プラットフォームに選ばせる |

> 💡 出典の資料の例では、ティアの選び方ひとつで同じタスクでも **約 24 倍のコスト差** が出ていた。typo 修正に Opus を出すのが一番ありがちなムダ。

### 2 — コンテキスト設計

> *「必要なだけ多く、必要以上には少なく」*

- **`/clear` をこまめに** — タスクが変わったら新しいセッション。前のコンテキストを引きずらない。
- **関連する情報だけを渡す** — ディレクトリ全体ではなく対象ファイルだけリンク。
- **コンパクト化** は有効だが情報が落ちる。慎重に。
- 詳しくは <a class="retro-link" href="/theomonfort/playbook/context-engineering">Context Engineering ↗</a> を参照。

### プロンプトは常時オン

プロンプト・システムメッセージ・ツール定義は **毎ループ** コンテキストウィンドウの末尾に居続ける。つまり：

- プロンプトを **トークン数で最適化しない**。**最初から正しい方向にエージェントを向ける** ことを最適化する。
- 50 トークンの曖昧なプロンプトで 30 ループ迷走するより、200 トークンの精密なプロンプトで 3 ループで着地するほうが桁違いに安い。

### ストップシグナルを書く

エージェントは指示がないと「次にやれそうなこと」を勝手に続ける。プロンプトに **どこで止まるか** を必ず入れる。

| 良くない | 良い |
| --- | --- |
| 「バグを直して」 | 「Issue #45 のバグを修正。**赤テストが緑になったら止まる**。commit / push はしない」 |

> 💡 ストップシグナルがないと、エージェントは勝手に commit → push → CI 完了待ち → 別のエラーを直す、まで突き進む。「止まる条件」を 1 行入れるだけで余計なループが消える。

## 分割統治：Research → Plan → Implement

調査・計画・実装を 1 セッションでやると、コンテキストが汚染される。**分けろ**。

| フェーズ | ゴール | 推奨モデル | 出力 |
| --- | --- | --- | --- |
| 🔍 **Research** | 「関係するファイルは？」 | Reasoning（Gemini 2.5 Pro, Opus 4.7） | ファイルと前提の短いリスト |
| 📋 **Plan** | 調査結果を精密な仕様に | Reasoning（Opus 4.7） | Markdown の計画 |
| 🛠️ **Implement** | 仕様を実行 | Mid ティア（Sonnet, GPT-5.4）または fleet | コード + テスト |

> 💡 <a class="retro-link" href="/theomonfort/playbook/cli">Copilot CLI ↗</a> の `/research`・`/plan`・`/fleet` がこの 3 フェーズに対応している。各フェーズは別のコンテキストウィンドウなので、肥大化した Research セッションが Implementer に届かない。

> ⚠️ **実装フェーズに Reasoning モデルを使わない。** 精密な仕様を渡しても、Reasoning モデルは「もっと良い設計があるのでは」と考え直して別の方向へ走ることがある。仕様を信じて実行してほしいフェーズでは、Mid ティアの方が **速く・安く・仕様通り** に動く。

## 決定的ガードレールが高価なリトライを防ぐ

テスト・リンター・型チェック・セキュリティスキャンは **トークン節約ツール** でもある。

| ガードレール無し | ガードレール有り |
| --- | --- |
| バグ修正 → バグ修正 2 → バグ修正 3 → デバッグセッション → インシデント | バグ修正 → **テストが即座に失敗** → 1 回の修正 → グリーン |

無いとエージェントは静かにバグを積み上げ、CI 時間・コードレビュー往復・人間のトリアージ時間で支払う羽目になる。**入った同じループでエラーが捕まる** ようにする。

> 📊 **証拠**: Copilot CLI チームのコードベースは **半分以上がテスト**。「決定的なガードレール」は理念ではなく、エージェントを安全に高速で走らせるための具体的な装備。

> 📘 <a class="retro-link" href="/theomonfort/playbook/hooks">Hooks ↗</a> と組み合わせて `pre-write → run-tests` を自動で強制すると効く。

## Agent Config 早見表

選ぶ Config によって **コンテキストに何がいつ入るか** が決まる。それぞれに品質とトークンの「指紋」がある：

| Config | いつ読み込まれる？ | トークン / 品質リスク | 向く用途 | 避けたいケース |
| --- | --- | --- | --- | --- |
| 📌 **永続インストラクション**（<a class="retro-link" href="/theomonfort/playbook/instructions">Instructions ↗</a>） | 常時・全ループ | 長過ぎると毎ループを膨らませる | プロジェクトの絶対ルール、エージェントの空振りログ、「簡潔に」など出力トリム（`copilot-instructions.md` か `AGENT.md` の **どちらか片方** だけ） | ドキュメント丸ごとや AI 生成の壁文字 |
| 🎭 **Custom Agent**（<a class="retro-link" href="/theomonfort/playbook/custom-agent">Custom Agent ↗</a>） | 明示的に呼び出したとき | 主に品質レバー。ツールセットも絞れる | ロール / ワークフローに **バイアスをかける**（`/tdd-red` など）、誤ったツール利用の予防 | 一発プロンプトで足りる挙動 |
| 🧩 **Skills**（<a class="retro-link" href="/theomonfort/playbook/agent-skills">Agent Skills ↗</a>） | LLM が必要と判断したとき遅延ロード | description だけでもトークンを食う。冗長な Skill が選択肢を埋める | モデルが本来持っていない能力 | モデルが既に得意なこと（例：「React Skill」） |
| 🔌 **MCP**（<a class="retro-link" href="/theomonfort/playbook/mcp">MCP ↗</a>） | 登録時点でツール定義が常時オンに | システムプロンプトを膨らませ、意図しない呼び出しを誘発しがち | 常に必要な外部 API | 重く / 状況依存のもの（Playwright 等）— Custom Agent でラップせよ |
| 🪞 **サブエージェント** | メインエージェント（または人間）が起動したとき | 追加トークンを消費するが、メインの汚染を防ぐ | リサーチ・コードベーススキャン。要約だけメインに返す | 些細な質問 — オーバーヘッドの方が高い |
| 📂 **スコープ付きインストラクション** | ファイルパスに応じて条件付きでロード | 永続と同じ。スコープが狭いだけ | フロント / バックなど明確に分かれたモノリス | 小さなリポ — 永続にまとめる |
| 📝 **プロンプトファイル** | 手動で呼び出し | ツールセットの絞り込みや Custom Agent 呼び出しに便利 | 既知ワークフローの再利用 | Copilot CLI（未対応）— Skill か Custom Agent を使う |
| 🧠 **Copilot Memory**（<a class="retro-link" href="/theomonfort/playbook/copilot-memory">Copilot Memory ↗</a>） | 自動、小さく常時オン | 通常は影響小だが、古い記憶は誤誘導する | リポ固有の知見を時間とともに蓄積させる | 最近の記憶を見直していないとき |

> ⚠️ 常時オンの Config はすべて **毎ループに乗る税金** と考える。トリム・無効化・スコープ化を惜しまない。

> 🧪 **永続インストラクションは「生きたドキュメント」。** Copilot CLI チームは四半期ごとに `copilot-instructions.md` を **白紙に戻す** 練習をする。コードベースもモデルも進化するので、古いエージェント空振りログは過剰指示になりがち。**書く → 運用する → 賞味期限を区切って捨てる** ループで保つ。AI に生成させない（人が書くのが契約）。「Caveman skill」のような出力トリム Skill は不要、`copilot-instructions.md` に「簡潔に」と一行入れれば足りる。

## 上級者向け — パワーユーザーのヒント

ここから先は条件付き＆トレードオフあり。上の基本を入れ終えてから手を出す。

- 🧮 **コードで考える** — 巨大な API レスポンスや長いファイルは、まずスクリプトでフィルタしてからエージェントに渡す。
- 🖥️ **CLI vs MCP** — `gh` / `kubectl` / `npm` などの CLI に頼る方が、同等の MCP より軽いことが多い（モデルが既に知っているから）。
- ✂️ **シェル出力をトリム** — <a href="https://github.com/rtk-ai/rtk" target="_blank" rel="noopener noreferrer" class="retro-link">rtk-ai/rtk</a> でうるさいコマンド出力を圧縮してからコンテキストに入れる。
- 📊 **`/chronicle tip` を定期的に** — Copilot CLI のセッションを分析し、改善点を具体的に教えてくれる。
- 🔁 **ツール呼び出しをまとめる** — <a href="https://github.com/jsturtevant/copilot-codeact-plugin" target="_blank" rel="noopener noreferrer" class="retro-link">copilot-codeact-plugin</a> は複数のツール呼び出しを 1 ラウンドトリップに畳む。
- 🎚️ **モデル別の細かいチューニング** — 可能だがモデルの進化が速いので、超大規模運用のみ価値あり。

## 長期的なマインドセット

- 🧭 **分析力を磨く。** 開発者の真の価値はコーディングではなく分析力とドメインへの素早い習熟だった。**ドメインの言葉でエージェントに正確に指示できる** ことが最も価値のあるスキルになる。
- 🏛️ **良いアーキテクチャを適用する。** DDD・ヘキサゴナル・CQRS・Event-Driven ── 境界が明確だとエージェントの空振りが減り、コードを誤った場所に置かなくなる。「5 行関数」議論は重要度を落とし、アーキテクチャの重要度は上がる。
- 🔧 **プロンプトと Config を反復する。** あなたはもう Context Engineer。エージェントの空振りはインシデント扱いし、Config を新鮮に保ち、`/chronicle` でパターンを探る。

## 今日から始められる 5 つ

1. ✅ **タスクに合ったモデルを選ぶ** ── typo 修正に Opus を出すのをやめる。
2. ✅ **プロンプトで明確に導く** ── まず方向付け、トリミングは後。
3. ✅ **Research → Plan → Implement** ── セッションを分け、フェーズごとにモデルを切り替える。
4. ✅ **決定的なガードレール** ── テスト・リンター・セキュリティスキャン。
5. ✅ **簡潔で人手で書かれた `copilot-instructions.md` を維持** ── エージェントの空振りログ＆出力トリム（「簡潔に」）として使う。

> 🎯 **1 つだけ覚えるなら：** **必要なだけ多く、必要以上には少なく** コンテキストを書く。コスト削減は品質に任せる。
