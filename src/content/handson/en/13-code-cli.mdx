**Copilot CLI** is an interactive AI assistant that runs in your terminal. Without ever opening VS Code, it can autonomously generate, review, and refactor code.

Here we'll take the **Issues you created at the end of Part 6** and have the CLI implement them **in parallel** via `/fleet`.

### Key commands

| Command       | Description                                            |
| ------------- | ------------------------------------------------------ |
| `/model`      | Pick the AI model (Claude / GPT, etc.)                 |
| `/mode`       | Switch mode (plan / agent)                             |
| `/allow-all`  | Allow tool execution without confirmation (Autopilot)  |
| `/fleet`      | Run multiple agents in parallel                        |
| `/tasks`      | Check the status of running fleet tasks                |
| `/chronicle`  | Analyze session history and usage                      |
| `/resume`     | Resume the previous session                            |

### 12.1 — Launch Copilot CLI

First, grab the latest code. The Cloud Agent and Code Review should have merged PRs by now, so update your local branch too:

```bash
git checkout main
git pull
```

Then launch Copilot CLI from the VS Code terminal:

```bash
copilot
```

<Aside type="negative">

**First launch**: If you're prompted with "Install GitHub Copilot CLI?", type `y` to install it. On Codespaces it's already pre-installed via `postCreateCommand`.

</Aside>

### 12.2 — Pick a model and enable Autopilot

Inside the CLI, run:

```
/allow-all
```

```
/model
```

Pick the most capable model available (e.g., Claude Opus 4.6).

Then press **Shift+Tab** to switch into **Agent (Autopilot) mode**.

### 12.3 — Implement Issues in parallel with `/fleet`

Have the CLI implement the 3–5 Issues you created in Part 6 in parallel. The prompt also includes an instruction to run a rubber-duck review with a different model (Codex family) at the end.

```
/fleet Take every open Issue in this repository labeled `bug` or `enhancement` and
assign one fleet agent per Issue to implement them in parallel.

Each agent should follow this flow:
1. Create a branch matching the Issue number (`fix/<number>-<slug>`)
2. Implement the fix
3. Verify it runs with `pnpm dev`
4. Commit and open a PR (include `Closes #<number>` in the body)

Once all PRs are open, switch to the **GPT-5 Codex** model at the end
(a different model from Claude, to avoid blind spots inherited from the implementer)
and read the diffs of every PR, reviewing them from a rubber-duck perspective.
Focus on: design-level blind spots and logic bugs, missed Issue requirements,
consistency with existing code, and missing tests / error handling.
Ignore style and trivial nits — list only the truly important issues, concisely, PR by PR.
```

⏱ ~10–15 minutes

The CLI spawns multiple fleet agents, each working on its own branch through implementation and PR creation autonomously. Progress streams in your terminal, and the rubber-duck review by the alternate model is printed at the end.

To check progress during and after execution:

```
/tasks
```

This lists the status of each fleet agent (running / done / failed) along with the current step.

<Aside type="positive">

**Key point**: Having a different model (Codex) review what another model (Claude) implemented helps you avoid same-model bias and surface blind spots.

</Aside>
