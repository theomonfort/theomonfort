### 6.1 — Switch the model to Claude Opus 4.8

Click the model selector at the bottom of the Copilot Chat panel and pick **Claude Opus 4.8**.

While you're there, also switch **"Default Approvals"** at the bottom of the panel to **"Bypass Approvals"**. Since we're working inside a Codespace sandbox, skipping the per-tool approval prompts is much faster.

<Aside type="positive">

**💡 Tip**: Opus-class models shine on complex design tasks and long-context conversations. For day-to-day editing and completion, other models are plenty, but for "scaffold the whole site in one shot" work like this section, Opus 4.8 gives more consistent quality.

</Aside>

### 6.2 — Switch to Plan mode

In Copilot Chat's mode switcher, select **"Plan"**.

<Aside type="positive">

**💡 Tip**: In Plan mode, Copilot focuses on **design and questions** rather than immediately editing files. It will push back on anything ambiguous in the requirements. Keep iterating with it until you're happy with the plan.

</Aside>

### 6.3 — Ask for an implementation plan

Copy the following prompt as-is and paste it into Copilot Chat:

````
Build the Playbook landing page in Japanese only (work on a new branch: `feature/playbook-landing-page`).

Requirements:
* Use `src/content/playbook/ja/*.md` as the source.
* Create `/playbook` with cards grouped by `category` (2 columns of cards).
* Use `src/lib/playbook-categories.ts` for category labels, actor names, avatars, descriptions, colors etc.
* Each card shows icon, title, summary, and uses the `accent` frontmatter for colors.
* Add decorative floating assets near the matching SDLC category (about 450px wide):
    * introduction → `public/planet-introduction.webp`
    * plan → `public/planet-plan.webp`
    * develop → `public/planet-develop.webp`
    * review → `public/planet-review.webp`
    * secure → `public/planet-secure.webp`
    * operate → `public/planet-operate.webp`
    * administration → `public/planet-administration.webp`
* Clicking a card opens a detail page that renders the full Markdown.
* Detail page has a “Back to Playbook” button.

Add a top “SDLC navigation chain” above the introduction section:
  Plan → Develop → Review → Test & Secure → Operate
  Each item should be a bordered pixel button using the matching category color, with arrows between items, and should link to the category anchor.

Once `npm run dev` confirms it works, commit and push your changes and open a pull request titled "Playbook site foundations".
````

Copilot may ask you a few questions (package manager, UI framework, deployment target, etc.). It should have read `copilot-instructions.md` and default to Astro / Tailwind / pnpm, but answer any gaps as they come up.

⏱ About 3 minutes

<Aside type="positive">

**Why it matters**: The Stack rule you wrote in 4.1 and the frontend design tokens from 4.2 pay off here — the plan should come back proposing Astro + Tailwind with the colors you specified.

</Aside>

### 6.4 — Implement with Autopilot

Once you're happy with the plan, click **"Start with Autopilot"** to kick off the implementation.

![Start with Autopilot](/theomonfort/handson/img/start-with-autopilot.png)

⏱ About 15 minutes

<Aside type="positive">

**💡 Work in parallel**: While Copilot is implementing, you've got time on your hands — hop over to [Step 7 (REVIEW: Copilot Code Review)](../review-copilot-code-review/) and enable **Automatic Copilot Code Review** now. That way, the moment Copilot opens the PR it's working on, the review fires automatically.

</Aside>

Once the implementation is done, start the dev server to check it out:

```bash
npm run dev
```

From the port-forwarding notification, click **"Open in browser"** and try it out.

<Aside type="positive">

**Try this**: Open `/playbook`, check that the category sections use the right colors, the cards are in two columns on desktop, and the planet assets feel large enough. Click a few tiles, confirm the detail pages render the full Markdown, and use the Back to Playbook button. Jot down anything that feels off, UI tweaks you'd like, or features you wish were there (we'll file them as Issues in the next step).

</Aside>

### 6.5 — File the improvements as Issues

Here's the key move — **instead of asking Copilot Chat to "fix this right now", file each improvement as a GitHub Issue**. In Step 11 (CODE: Cloud Agent) we'll have Cloud Agent implement some of them **in parallel, in one batch**, and in Step 13 (CODE: Copilot CLI) we'll knock out the remaining ones in parallel with `/fleet`.

Send the prompt below to Copilot Chat to file **8** Issues at once (the `github-issues` skill will structure them automatically):

<Aside type="reference">
The Issues below are **just one example** of the improvements you might want, based on how Copilot built the site. Depending on what Copilot actually shipped, some of them may not apply — skip those, and **feel free to swap in your own improvements** that make the site better the way you want.
</Aside>

```
Use the github-issues skill to create the following improvements as individual Issues in this repository.

1. Title: Add category descriptions and actors to the Playbook landing page
   Details: Use `src/lib/playbook-categories.ts` to show each category's actor avatar, actor name, and description above that category's cards.

2. Title: Tune planet placement and sizing
   Details: Make each `planet-*.webp` feel like a large decorative background element. It should be visible, but not cover text or block clicks. Check desktop and mobile separately.

3. Title: Add active styling to the SDLC navigation chain
   Details: The top Plan → Develop → Review → Test & Secure → Operate buttons should use matching category colors and have hover/focus states. If possible, highlight the category currently in view.

4. Title: Improve Playbook detail page typography
   Details: The detail page should render headings, tables, lists, code blocks, blockquotes, and links with the same retro theme as the landing page.

5. Title: Render Mermaid diagrams in detail pages
   Details: If a Markdown detail page contains a ` ```mermaid ` block, render it as a Mermaid diagram instead of plain code.

6. Title: [Bug] Public asset paths fail under the `/theomonfort/` base path
   Details: Icons, planets, and images should load correctly when the site is served from `/theomonfort/`, not only from `/`.

7. Title: [Bug] Mobile layout has overlapping planets or cramped cards
   Details: On narrow screens, cards should be one column, planets should shrink or move behind sections, and text should remain readable.

8. Title: Add a simple search or filter for Playbook cards
   Details: Add a small client-side search box that filters cards by title and summary while keeping category grouping understandable.
```

You can find the resulting Issues in the Issues tab of your repository. We'll have Cloud Agent / Copilot CLI implement them in the next sections.

⏱ About 3 minutes
